# 🎯 Git 開發者技術水平分析系統

## 📋 專案概述

本專案提供完整的 Git 版控數據分析工具，協助評估開發者的技術水平和程式碼品質。

---

## 🗂️ 檔案結構

```
Lab.PersonInfo/
├── README.md                      # 使用指南
├── PROJECT_SUMMARY.md            # 專案總覽（本檔案）
│
├── 📊 分析工具
│   ├── analyze_developer.sh       # Shell 快速分析腳本
│   ├── developer_analyzer.py      # Python 完整分析工具
│   ├── analyze_all_developers.sh  # 批次分析所有開發者
│   └── progress_analyzer.py       # 進步趨勢對比分析
│
└── 📁 output/                     # 輸出報告目錄
    ├── 評估框架說明.md            # 評估維度詳細說明
    ├── 完整評估指南.md            # 完整使用指南
    ├── [開發者]_YYYYMMDD.md       # 個人詳細報告
    ├── [開發者]_progress_YYYYMMDD.md  # 進步分析報告
    └── summary_YYYYMMDD.md        # 團隊匯總報告
```

---

## 🚀 快速開始

### 1. 分析單一開發者

```bash
# 方法一：快速統計（Shell）
./analyze_developer.sh "開發者名稱" "2024-01-01" "2024-12-31"

# 方法二：完整報告（Python）
python3 developer_analyzer.py "開發者名稱" "2024-01-01" "2024-12-31"
```

### 2. 批次分析整個團隊

```bash
./analyze_all_developers.sh "2024-01-01" "2024-12-31"
```

### 3. 分析進步趨勢

```bash
python3 progress_analyzer.py "開發者" "2024-01-01" "2024-06-30" "2024-07-01" "2024-12-31"
```

## 🎯 核心問題解答

### ❓ 如何從版控知道開發者的技術水平？

**回答：** 透過 6 大維度綜合評估：

1. **貢獻量** - 看活躍度和產出（提交次數、程式碼量）
2. **品質** - 看 commit message 規範、變更粒度、修復率
3. **技術廣度** - 看涉及的語言和技術棧
4. **協作** - 看 merge、conflict、revert 情況
5. **工作模式** - 看時間分佈的規律性
6. **進步** - 看不同時期的對比變化

### ❓ 如何從版控知道程式碼品質？

**直接指標（Git 可獲得）：**
- ✅ Commit message 規範性（Conventional Commits）
- ✅ 變更粒度（小型 commit 佔比）
- ✅ 修復率（fix/bug 相關提交比例）
- ✅ Revert 頻率（被回退次數）

**間接指標（需結合工具）：**
- ⚠️ 測試覆蓋率變化
- ⚠️ 重構行為
- ⚠️ Code Review 參與度

## 🛠️ 工具詳細說明

### 1. analyze_developer.sh
- **類型：** Bash 腳本
- **用途：** 終端機快速統計
- **優點：** 無依賴、輕量、快速
- **輸出：** 純文字統計

### 2. developer_analyzer.py
- **類型：** Python 3 腳本
- **用途：** 生成完整 Markdown 報告
- **優點：** 深度分析、自動評分、格式化輸出
- **輸出：** `output/[開發者]_YYYYMMDD.md`

**包含內容：**
- 📊 貢獻統計
- ✅ Commit 品質分析
- 🔧 技術棧分析
- ⏰ 工作模式分析
- 🎯 綜合評估與評分
- 💡 改進建議

### 3. analyze_all_developers.sh
- **類型：** Bash 批次腳本
- **用途：** 分析所有開發者並生成匯總
- **輸出：** 
  - 各開發者個別報告
  - 團隊匯總表格

### 4. progress_analyzer.py
- **類型：** Python 3 腳本
- **用途：** 對比兩個時間段的進步情況
- **輸出：** `output/[開發者]_progress_YYYYMMDD.md`

**包含內容：**
- 📅 時間段對比表
- ✅ 品質指標變化
- 🎯 進步/退步分析
- 💡 發展建議

## ⚠️ 重要提醒

### 評估局限性

**無法評估的能力：**
- ❌ 程式碼邏輯品質（需 Code Review）
- ❌ 演算法效率（需效能測試）
- ❌ 安全意識（需安全掃描）
- ❌ 架構設計能力（需設計文檔）
- ❌ 業務理解（需需求文檔）

**可能失真的情況：**
- ⚠️ Squash merge 隱藏提交歷史
- ⚠️ Pair programming 歸屬不明
- ⚠️ 接手遺留專案有大量刪除
- ⚠️ 使用生成工具灌水

### 正確使用方式

✅ **建議做法：**
- 作為初步篩選工具
- 觀察長期趨勢
- 結合 Code Review
- 參考業務成果

❌ **避免做法：**
- 單純依據分數評估
- 作為 KPI 考核
- 忽略業務價值
- 讓開發者為指標優化

## 📞 常見問題

**Q: 為什麼我的評分這麼低？**  
A: 最常見原因是 Commit Message 不規範（佔 25% 權重）。立即採用 Conventional Commits 可顯著提升。

**Q: 提交次數少會影響評分嗎？**  
A: 會，但影響不大（僅 15%）。品質比數量重要。

**Q: 如何快速提升評分？**  
A:
1. 規範 commit message（立即見效）
2. 拆分大型 commit（中期見效）
3. 減少 bug 修復比例（長期見效）
