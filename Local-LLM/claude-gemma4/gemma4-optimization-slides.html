核心配置

硬核設定：防迴圈與 32K 上下文配置

1. Modelfile 配置

FROM gemma4:e4b
# 手動強制擴展上下文至 32k
PARAMETER num_ctx 32768
# 設定確定性輸出以確保 Tool Calling 格式
PARAMETER temperature 0.0
# 關鍵：防止模型在長上下文下陷入重複詞彙
PARAMETER repeat_penalty 1.15

2. litellm_config.yaml 配置

model_list:
  - model_name: gemma4-opt
    litellm_params:
      model: ollama_chat/gemma4-opt
      api_base: http://localhost:11435
      extra_body:
        think: false # 關閉原生思考，省43% tokens
litellm_settings:
  drop_params: true # 過濾不相容 API 參數

避坑指南

技術落地：四大踩坑回顧與對策

1. 模型架構量化坑

8GB 顯示記憶體下，gemma4:e4b 是唯一選擇。切勿使用 12B dense 模型加 32K KV Cache，這將直接導致 CUDA out of memory (OOM)。

2. 參數報錯(400)坑

Claude Code 會主動發送 reasoning_effort。LiteLLM 必須配置 drop_params: true 過濾掉不相容參數，避免 API 傳回 400 Bad Request。

3. 長對話重複輸出坑

Gemma4 在大 context 下容易發生「您 I I I」死迴圈。必須在 Modelfile 中指定 PARAMETER repeat_penalty 1.15 進行懲罰壓制。

4. 原生思考相容坑

為避免 gemma4 原生 thinking block 消耗額外 43% token 並造成解析錯誤，應在 LiteLLM 層加入 think: false 停用思考推理輸出。
